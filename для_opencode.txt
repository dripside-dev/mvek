● I found the bug. In restore(), the floor is regenerated fresh from seed —
  which re-locks every door that was originally a locked-room door (the floor
  generator's default). The save snapshot only records doors that were still 
  locked at save time (runsave.py:47). On reload, restore only ever sets doors
  to True (line 110-113) and never sets a door back to False. So a door you
  already paid a key to unlock gets re-locked on reload — and if you have no
  key, you're trapped.

  The fix: apply the saved locked-state authoritatively — every door not in the
  saved list must be reset to False, not left at the regenerated True default.

     Портирование MVEK на HTML5 (pygbag)

     Context

     Игра — pygame-приложение, запускается нативно python main.py. Нужно
     перенести её на сайт как
     HTML5/WASM. Стандартный инструмент — pygbag (компилирует pygame-ce в
     WebAssembly). Чтобы игра
     работала в браузере, нужны три вещи:

     1. Кооперативный async-цикл. Сейчас главный цикл — синхронный while True
     (mvek/main.py:2402),
     а выход — pygame.quit(); sys.exit(0) (строки ~2016 и ~2145). В браузере
     поток нельзя блокировать:
     каждый кадр должен отдавать управление через await asyncio.sleep(0), а
     «выход» не должен звать
     sys.exit.
     2. Точка входа для pygbag — корневой main.py с asyncio.run(async main()).
     3. Постоянные сейвы. Виртуальная ФС pygbag сбрасывается при перезагрузке
     вкладки. Пользователь
     выбрал персистентность → пишем mvek_save.json в localStorage браузера.

     отсутствии numpy просто
     молчат; музыка идёт через pygame.mixer.music (.mp3), что pygbag
